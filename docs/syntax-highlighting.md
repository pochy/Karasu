# シンタックスハイライト選定

Karasu の Markdown プレビューにおけるコードブロックのシンタックスハイライトについて、候補の比較と採用方針をまとめる。

## 採用ライブラリ

**[sugar-high](https://github.com/huozhi/sugar-high)** を採用する。

実装は `src/markdown.ts` で `marked` の `code` レンダラーに組み込み、プレビュー表示時のみ実行する。

## 評価の前提

Karasu では次を優先する。

| 優先度 | 観点 | 説明 |
|--------|------|------|
| 高 | **起動中の CPU** | プレビュー切替・再描画時のメインスレッド負荷 |
| 高 | **起動中のメモリ** | 常駐する文法・WASM・登録言語データ |
| 高 | **GPU / 描画** | ハイライト後の DOM ノード数とスクロール性能 |
| 中 | **I/O** | 初回ロード時のネットワーク・ディスク（フォント等） |
| 低 | **ビルドサイズ** | バンドルや配布物の KB 数（本プロジェクトでは最優先にしない） |

設計書（`docs/superpowers/specs/2026-06-03-tauri-markdown-viewer-design.md`）の「プレビューは必要時に更新する」「重い依存を増やしすぎない」とも整合する。

## 候補の比較

### 一覧

| ライブラリ | ビルドサイズ（目安） | 起動中メモリ | 起動中 CPU | GPU / DOM | 言語カバー | 備考 |
|------------|----------------------|--------------|------------|-----------|------------|------|
| **sugar-high** | 極小（〜1 KB gzip） | **小** | プレビュー時のみ・同期 | span 数は比較的抑えめ | JS/TS/JSX + プリセット数種 | **採用** |
| lowlight | 中（common で 20〜40 KB 程度） | 登録言語に比例 | 同期・highlight.js 系 | 中〜大 | 多（絞れる） | 必要言語だけ register なら現実的 |
| Prism（最小構成） | 中 | 中 | 同期 | 中 | 中 | 定番・テーマ豊富 |
| highlight.js（全言語） | 大（500 KB 超になりがち） | **大** | 同期 | 大 | 非常に多 | 常駐が重くなりやすい |
| **Shiki** | 大 | **大**（WASM + 文法） | 初回・変換ともに重め | 高品質だが DOM も重い | 非常に多 | 品質最高だがランタイム向きではない |
| starry-night | 大 | 大 | 重め | — | 多 | 精度重視・常駐コスト高 |

### ビルドサイズだけ見た場合（参考）

以前の整理では「軽量」軸で sugar-high が有利だった。本プロジェクトでは **ビルドサイズは決め手にしない** が、sugar-high はランタイム面でも有利なことが多い。

### ランタイム重視で見た場合（本プロジェクトの主軸）

#### sugar-high（採用）

- WASM や巨大な文法テーブルを常駐しない。
- 初期化コストが小さく、プレビュー初回の体感が軽い。
- ハイライトはプレビュー切替時に限れば、編集中の CPU はゼロ。
- トークン色は CSS 変数（`--sh-*`）で指定でき、既存のダークコードブロックと統合しやすい。

#### lowlight / Prism

- 使うなら **必要な言語だけ register** することが必須。全言語登録はメモリ的に最悪に近い。
- ハイライト処理は基本的にメインスレッド同期。巨大な MD ではプレビュー瞬間に CPU スパイク。
- sugar-high より言語は増やしやすいが、常駐・DOM ともに重くなりがち。

#### Shiki（不採用）

- ビルドが大きいだけでなく、**ランタイムでも WASM・文法・テーマが常駐**しやすい。
- 初回プレビューでコンパイル・ウォームアップによる CPU スパイクが起きうる。
- 見た目は最高クラスだが、「起動中のメモリ・CPU・GPU を抑える」という Karasu の方針と相性が悪い。

## Karasu での実装方針

### いつハイライトするか

- **編集画面**: ハイライトしない（入力遅延ゼロ）。
- **プレビュー画面**: `marked` 変換時にフェンス付きコードブロックのみ `sugar-high` で HTML 化。
- **キャッシュ**: 内容が変わっていない限り、編集 ↔ プレビューの往復では再パース・再ハイライトしない（`src/markdown.ts` の `previewCache`）。

### 対応言語（プリセット）

フェンスの言語ラベルから次のプリセットを選択する。未対応・空は JS/TS 系のデフォルトルール。

| ラベル例 | プリセット |
|----------|------------|
| `rust`, `rs` | rust |
| `python`, `py` | python |
| `css`, `scss`, `sass`, `less` | css |
| `go`, `golang` | go |
| `java` | java |
| `c`, `cpp`, `h`, `hpp` | c |
| `diff` | diff |
| `javascript`, `typescript`, `js`, `ts`, `jsx` 等 | デフォルト（JS/TS） |

### スタイル

プレビュー内の `pre` に sugar-high 用 CSS 変数を定義する（`src/styles.css`）。コードブロック背景は DADS 風のダーク（`#1e1e1e`）のまま、トークン色のみハイライトする。

### 将来の拡張（必要になった場合）

- **超大きいファイル**: コードブロックのハイライトをスキップする、またはブロック数上限。
- **UI が固まる場合**: Worker へハイライト処理を移す（メモリはワーカー分増えるがメインスレッドの CPU 占有は下がる）。
- **言語追加**: `sugar-high/presets` にあるものは `src/markdown.ts` の `LANG_PRESETS` にマッピングを足す。足りない場合のみ lowlight 等への乗り換えを再検討。
