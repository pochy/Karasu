# Karasu 設計判断の記録

このドキュメントは、Karasu が **なぜ今の形になっているか** を他の開発者に伝えるためのものです。  
機能一覧だけでなく、検討した候補・採用しなかった案・トレードオフを残します。

初期の要件定義は [`superpowers/specs/2026-06-03-tauri-markdown-viewer-design.md`](superpowers/specs/2026-06-03-tauri-markdown-viewer-design.md) を参照してください。本書はその後の実装で追加・変更した部分を中心に説明します。

拡張案の評価表・実装フェーズ・却下理由の一覧は [機能バックログ（提案一覧）](#機能バックログ提案一覧) を参照。

## 設計で最優先していること

Karasu の価値は「Markdown を軽く編集・閲覧する」ことです。性能や依存の議論では次の順で判断しています。

| 優先度 | 観点               | 意味                                                           |
| ------ | ------------------ | -------------------------------------------------------------- |
| 1      | **起動中の CPU**   | 編集中・待機中にメインスレッドや WebView が忙しくならないこと  |
| 2      | **起動中のメモリ** | 常駐する文法・WASM・巨大ツリー・全フォント読み込みを避けること |
| 3      | **GPU / 描画**     | 不要な DOM や再描画でスクロールが重くならないこと              |
| 4      | **I/O**            | 起動や操作のたびにディスク・ネットワークを叩きすぎないこと     |
| 5      | **ビルドサイズ**   | 配布物の KB は参考程度（軽さの主戦場ではない）                 |

「ビルドが小さい＝ランタイムが軽い」ではない（例: Shiki はビルドもランタイムも重くなりやすい）。

## アーキテクチャ概要

```
┌─────────────────────────────────────────────────────────┐
│  WebView（フロント: TypeScript + CSS）                    │
│  ├ ツールバー（開く / 別名で保存 / 一覧 / プレビュー / 設定） │
│  ├ サイドバー（任意表示）… ファイルツリー                  │
│  ├ エディタ（textarea）                                   │
│  └ プレビュー（marked GFM + Mermaid 遅延描画、sugar-high）  │
└───────────────────────────┬───────────────────────────────┘
                            │ Tauri invoke（必要時のみ）
┌───────────────────────────▼───────────────────────────────┐
│  Rust                                                     │
│  ├ read_file / write_file                                 │
│  ├ get_recent_path / get_recent_paths（MRU、最大 10 件）   │
│  ├ get/set_workspace_root（作業フォルダ）                  │
│  ├ list_directory（1 階層ずつ）                            │
│  ├ search_filenames（作業フォルダ内ファイル名検索）        │
│  ├ set_workspace_watch（notify、設定 ON 時のみ）           │
│  └ list_system_fonts（設定画面用）                         │
└───────────────────────────────────────────────────────────┘
```

**意図**: 重い処理は Rust に寄せつつ、**編集ループはフロントのテキストと CSS だけ**で回す。プレビュー HTML 生成は「プレビューに切り替えたとき」に限定する。

## 技術スタック

| 層                   | 採用                             | 候補・却下                                   | 理由                                                                               |
| -------------------- | -------------------------------- | -------------------------------------------- | ---------------------------------------------------------------------------------- |
| デスクトップ         | **Tauri 2**                      | Electron                                     | ネイティブ WebView でメモリ・起動が比較的軽い。Markdown ビューワー程度の要件に十分 |
| フロント             | **Vite + TypeScript（Vanilla）** | React / Vue                                  | 状態が単純。フレームワーク常駐コストを避ける                                       |
| Markdown             | **marked + GFM 拡張 + Mermaid**  | micromark + 自作パイプライン                 | `marked-alert` / `marked-footnote` / `marked-gfm-heading-id`。Mermaid は `import()` で別チャンク・プレビュー時のみ |
| ネイティブダイアログ | **tauri-plugin-dialog**          | Rust `rfd` のみ                              | JS から「開く」操作と統合しやすい。常時ポーリングはしない                          |
| Rust プラグイン      | **dialog + opener**              | fs / shell / 多数プラグイン                  | 外部リンクはプレビュー時のみ。ファイル I/O は自前コマンド                          |

## 画面構成と UX

### 編集とプレビューは「切り替え」、常時 2 ペインではない

| 候補                                      | 採用     | 理由                                                                                                        |
| ----------------------------------------- | -------- | ----------------------------------------------------------------------------------------------------------- |
| 左右分割（エディタ + プレビュー常時表示） | 却下     | DOM とプレビュー更新が常に走り、GPU・CPU 負荷が増える。初期設計の「複数ペインの常時表示は含めない」にも合致 |
| **1 画面で編集 ↔ プレビュー切替**         | **採用** | `Cmd/Ctrl+P`。プレビューは必要時だけ `marked` + ハイライト                                                  |

### 保存は明示のみ

| 候補                                   | 採用     | 理由                                                    |
| -------------------------------------- | -------- | ------------------------------------------------------- |
| 自動保存                               | 却下     | 意図しない上書きを避ける。設計書どおり                  |
| **`Cmd/Ctrl+S`（上書き）+ 別名で保存** | **採用** | 保存状態を「保存済み / 未保存」で明示。自動保存はしない |

### メニューバー常駐（macOS トレイ）

| 候補 | 採用 | 理由 |
| ---- | ---- | ---- |
| ウィンドウを閉じるとプロセス終了 | 却下 | バックグラウンドから素早く再表示したい |
| **閉じる = 非表示 + 全リソース解放** | **採用** | 赤ボタンで `hide`。CSV セッション・監視・エディタ内容を破棄し、トレイアイコンのみ残す。待機 CPU はほぼゼロ |
| トレイから作業フォルダ / 最近のファイル | **採用** | ウィンドウ非表示中も Rust 側ネイティブメニューで操作可能 |
| Dock アイコンのみ残してメニューバー非表示 | 却下 | macOS ではメニューバー常駐が目的。`LSUIElement` 化は未採用 |
| 閉じる = WebView 破棄・再生成 | 却下 | 再表示のたびに起動コスト・ちらつき。`hide` + `suspend` で十分 |

**詳細（正本）**: [メニューバー常駐とリソース解放（G1）](#メニューバー常駐とリソース解放g1)

### プレビューの見た目（DADS 風）

| 候補                                       | 採用                | 理由                                                                                                           |
| ------------------------------------------ | ------------------- | -------------------------------------------------------------------------------------------------------------- |
| エディタと同じダークテーマ                 | 却下（プレビュー）  | 長文の読みやすさのためプレビューは常にライト背景                                                               |
| **デジタル庁 DADS 系のタイポ**（参考 CSS） | **採用**            | 見出し階層・行間・コードブロック（ダーク）を文書向けに整理。VS Code の Markdown プレビュー用テーマを参考にした |
| Google Fonts（Noto）                       | 一度導入後 **削除** | オンライン I/O と常駐フォントキャッシュを避け、システムフォント方針に統一                                      |

## メニューバー常駐とリソース解放（G1） {#メニューバー常駐とリソース解放g1}

Karasu は **プロセスを終了せずメニューバー（システムトレイ）に常駐** しつつ、**ウィンドウを閉じた瞬間に編集関連のメモリ・I/O・バックグラウンド処理を可能な限り解放** する。  
CSV セッション・索引スレッド・RSS の挙動の詳細は [`csv-spreadsheet.md`](csv-spreadsheet.md) の「メモリ管理と解放」を参照。本節は **アプリ全体のライフサイクル** の正本です。

### 目的

| 項目 | 内容 |
| ---- | ---- |
| ユーザー体験 | 赤ボタンで「終了」ではなく「しまう」。メニューバーからすぐ再開 |
| 性能 | 待機中は **CPU・I/O をほぼゼロ** に近づける（監視停止、索引スレッド停止、DOM/バッファクリア） |
| スコープ | 常駐型ユーティリティではない。**編集に必要なときだけ** WebView 内リソースを載せる |

### 用語

| 用語 | 意味 |
| ---- | ---- |
| **トレイ / メニューバーアイコン** | macOS 画面上部の Status Item 相当。Tauri 2 の `TrayIcon` |
| **suspend（サスペンド）** | ウィンドウ非表示前にフロント各エディタが行う **明示的な状態破棄** |
| **`suspend_app_resources`** | Rust 側一括解放 Tauri コマンド |
| **hide** | メインウィンドウを非表示にするだけ。プロセスと WebView は残る |

### アーキテクチャ概要

```
┌─────────────────────────────────────────────────────────────────────┐
│  メニューバー（Rust: tray.rs）                                       │
│  TrayIcon + ネイティブ Menu（表示 / 作業フォルダ / 最近 / 終了）    │
│  ※ 待機中も常駐。JavaScript は動かさない                             │
└───────────────────────────────┬─────────────────────────────────────┘
                                │ emit / dialog（必要時のみ）
┌───────────────────────────────▼─────────────────────────────────────┐
│  WebView（非表示時は suspend 済み・空に近い状態）                      │
│  lifecycle.ts … CloseRequested → suspendAll → invoke → hide         │
│  markdown / json / csv … EditorController.suspend()                 │
└───────────────────────────────┬─────────────────────────────────────┘
                                │ invoke
┌───────────────────────────────▼─────────────────────────────────────┐
│  Rust                                                                │
│  lifecycle.rs … close_all_sessions + 監視停止 + trim_process_heap   │
│  csv/session.rs … invalidate_all_sessions（索引スレッドも停止）      │
└─────────────────────────────────────────────────────────────────────┘
```

**レイヤ分担**

| レイヤ | 常駐するもの | ウィンドウ close 後 |
| ------ | ------------ | ------------------- |
| トレイ（Rust） | アイコン + メニュー定義 | 同左（軽量） |
| WebView | プロセス本体 | 空エディタ・空 DOM（suspend 後） |
| Rust CSV | — | セッション 0 件 |
| Rust 監視 | — | `notify` watcher なし |

### ウィンドウを閉じたとき（ユーザー操作）

赤ボタン・`Cmd+W` 等 → `CloseRequested` を **フロントで intercept**（`event.preventDefault()`）。

```
CloseRequested
  ↓ いずれかのエディタが dirty？
  │   yes → confirm（破棄して閉じるか）
  │   no  → 続行
  ↓ suspendAll() … markdown / json / csv の suspend() を順に await
  ↓ invoke("suspend_app_resources") … Rust 一括解放
  ↓ invoke("refresh_tray_menu") … 最近ファイルサブメニュー更新
  ↓ getCurrentWindow().hide()
  ↓ プロセスは生きたまま。Dock アイコンも残る（通常の Tauri アプリ）
```

確認ダイアログ文言（`src/app/lifecycle.ts`）:

> 未保存の変更があります。ウィンドウを閉じてメモリを解放しますか？  
> アプリはメニューバーに残ります。

### `EditorController.suspend()`（フロント）

`deactivate()`（モード切替）とは別。**ウィンドウ非表示専用** で、開いているファイル状態も破棄する。

| エディタ | `suspend()` の主な処理 |
| -------- | ---------------------- |
| **Markdown** | スクロール位置を `localStorage` へ保存 → 監視 OFF → エディタ/プレビュー文字列クリア → `invalidatePreviewCache()` → サイドバー `suspend()` |
| **JSON** | スクロール保存 → エディタ/ツリー DOM クリア → path / content リセット |
| **CSV** | `releaseBackend()`（`sheet.close` + `csvClose`）→ path / meta / dirty クリア |

#### サイドバー `suspend()` / `restoreFromDisk()`

| メソッド | タイミング | 処理 |
| -------- | ---------- | ---- |
| `suspend()` | ウィンドウ close | ツリー/最近/検索 DOM クリア、`cache`/`expanded`/`workspaceRoot` をメモリから削除、監視 OFF。**`workspace.json`（ディスク）は削除しない** |
| `restoreFromDisk()` | Markdown `activate()` で `getWorkspaceRoot() === null` のとき | ディスクから作業フォルダを再読込、ツリー再描画 |

### `suspend_app_resources`（Rust）

Tauri コマンド。フロントの `suspendAll()` **完了後** に呼ぶ。

```rust
pub fn suspend_app(app: &AppHandle, csv: &CsvState) -> Result<(), String> {
    close_all_sessions(csv)?;                        // CSV 全セッション + 索引スレッド無効化
    watch::set_workspace_watch(app, false, None)?;   // notify 停止
    memory::trim_process_heap();                     // macOS: malloc_zone_pressure_relief
    Ok(())
}
```

| 処理 | 詳細 |
| ---- | ---- |
| `close_all_sessions` | `invalidate_all_sessions` → `SESSION_EPOCH` インクリメント、行データ/LRU/索引チェックポイント破棄。[`csv-spreadsheet.md`](csv-spreadsheet.md) 参照 |
| 監視停止 | バックグラウンド debounce スレッドも watcher drop で終了 |
| `trim_process_heap` | OS へのヒープ返却を**促す**（RSS 即減を保証しない） |

### トレイメニュー（Rust `tray.rs`）

起動時 `setup_tray` で `TrayIconBuilder` を構築。`src-tauri/Cargo.toml` に `tauri` feature **`tray-icon`** が必要。

| メニュー項目 | ID | 動作 |
| ------------ | -- | ---- |
| **Karasu を表示** | `tray.show` | `main` ウィンドウを `unminimize` → `show` → `set_focus` |
| **作業フォルダを開く…** | `tray.open_workspace` | Rust `blocking_pick_folder` → `workspace.json` 保存 → `tray-workspace-opened` emit → ウィンドウ表示 |
| **最近のファイル** | `tray.recent.0` … | `recent.json` から最大 10 件。ファイル名のみ表示（40 文字超は省略） |
| **終了** | `tray.quit` | `app.exit(0)` |

**トレイ操作**

| 操作 | 挙動 |
| ---- | ---- |
| 左クリック | メニュー表示（`show_menu_on_left_click: true`）。**Click Up の直前に** `refresh_tray_menu` で最近リストを最新化 |
| 左ダブルクリック | メニューを開かずウィンドウ表示 |
| アイコン | `icon_as_template(true)` … macOS ライト/ダークメニューバー向け |

**ウィンドウ非表示中の作業フォルダ選択**  
フロントの `@tauri-apps/plugin-dialog` ではなく **Rust 側 `tauri-plugin-dialog`** を使う。WebView を前面表示しなくてもフォルダ選択できる。

### Tauri イベント（フロントが subscribe）

| イベント | payload | 発火元 | フロントの処理 |
| -------- | ------- | ------ | -------------- |
| `tray-open-file` | `string`（パス） | トレイ「最近」選択 | `editorModeForPath` → モード切替 → `openFileWithGuard` |
| `tray-workspace-opened` | `string`（フォルダパス） | トレイ「作業フォルダ」 | サイドバーが root 更新・ツリー再描画 |
| `tray-error` | `string` | Rust（パス取得失敗等） | `host.showError` |

拡張子 → モード判定: `src/core/file-kind.ts` の `editorModeForPath()`。

### ウィンドウ再表示時

| 操作 | 結果 |
| ---- | ---- |
| トレイ「Karasu を表示」 / ダブルクリック | **空の UI**（ファイル未選択）。起動直後に近い |
| トレイ「最近のファイル」 | 該当モードに切替 → ファイルを開く |
| トレイ「作業フォルダ」 | Markdown サイドバーにフォルダ反映（ファイルは自動で開かない） |

**起動時 `restoreRecentOnStartup` との違い**

| タイミング | 最近ファイル自動オープン |
| ---------- | ------------------------ |
| 初回 `DOMContentLoaded` | 各モードの `restoreRecentOnStartup`（CSV は CSV モード時のみ） |
| トレイから hide → show | **自動オープンしない**（suspend で状態クリア済み） |

### `deactivate()` / CSV「解放」との違い

| 操作 | ウィンドウ | Rust CSV | 未保存 (dirty) |
| ---- | ---------- | -------- | -------------- |
| **モード切替** `deactivate()` | 表示のまま | CSV: dirty ならセッション保持 | Markdown/JSON: UI のみ |
| **CSV「解放」ボタン** | 表示のまま | 全セッション破棄 | confirm 後破棄 |
| **ウィンドウ close → suspend** | **hide** | 全セッション破棄 | confirm 後破棄 |
| **トレイ「終了」** | — | プロセス終了 | — |

### 待機時のパフォーマンス

ウィンドウ非表示 + suspend 完了後、**意図的に動かさないもの**:

| 対象 | 状態 |
| ---- | ---- |
| ファイル監視 (`notify`) | 停止 |
| CSV 索引バックグラウンドスレッド | `SESSION_EPOCH` でキャンセル |
| TanStack Virtual prefetch / scroll リスナー | `sheet.close()` で解除 |
| Mermaid / marked パース | プレビュー DOM 空のため走らない |
| サイドバー `workspace-files-changed` | 監視 OFF のため emit なし |

**常駐コスト（許容）**

| 対象 | 理由 |
| ---- | ---- |
| WebView プロセス | Tauri 構成上、完全終了しない限り残る |
| トレイアイコン + メニュー | 再表示のため Rust 側に常駐 |
| `localStorage` / `workspace.json` / `recent.json` | ディスク上の設定のみ。メモリ上の作業データではない |

### Activity Monitor / RSS について

- **suspend 後も RSS が大きく減らない** ことはある。WebView ヒープ + macOS malloc キャッシュが主因。
- **論理的な解放**（CSV セッションなし・監視なし・エディタ文字列空）と **RSS の数値** は一致しないことが多い。
- CSV 固有の FAQ: [`csv-spreadsheet.md` — メモリ管理と解放](csv-spreadsheet.md#メモリ管理と解放)

### Tauri コマンド（G1 で追加）

| コマンド | 引数 | 説明 |
| -------- | ---- | ---- |
| `suspend_app_resources` | なし | CSV 全 close + 監視停止 + heap trim |
| `refresh_tray_menu` | なし | トレイの「最近のファイル」サブメニューを `recent.json` から再構築 |

### Tauri capabilities（フロント API と権限の対応）

`core:default` に含まれるのは **読取系ウィンドウ API**（`is-visible`, `title` 取得など）と **event listen** まで。変更系は **個別に allow が必要**。

| フロントの利用 | ファイル | 必要な permission | 備考 |
| -------------- | -------- | ----------------- | ---- |
| `getCurrentWindow().hide()` | `lifecycle.ts` | `core:window:allow-hide` | 無いと赤ボタン後にウィンドウが残る |
| `onCloseRequested` | `lifecycle.ts` | （`core:event:default` で可） | `listen` のみ。`preventDefault` は JS 側 |
| `getCurrentWebviewWindow().setTitle()` | `chrome.ts` | `core:window:allow-set-title` | 未保存 `•` 付きタイトル。無いと常に "Karasu" のまま |
| `listen("tray-open-file")` 等 | `lifecycle.ts`, `sidebar.ts` | `core:event:default` | 含まれる |
| `invoke(カスタムコマンド)` | 各所 | **local capability なら追加不要** | `capabilities/default.json` の `"local": true` |
| `open` / `save`（dialog） | 各エディタ | `dialog:default` | 含む |
| `openUrl` | `markdown/editor.ts` | `opener:default` | http(s) のみ |
| トレイ構築・`window.show` | `tray.rs`（Rust） | フロント ACL 不要 | ネイティブ API |

`allow-show` / `allow-unminimize` / `allow-set-focus` は **Rust トレイからの再表示** が主だが、将来フロントから `show()` する場合にも必要。

**カスタム invoke**（`read_file`, `csv_open`, `suspend_app_resources` 等）は `generate_handler` 登録 + local capability により **明示列挙なしで許可**（`app-manifest` の permission files は空）。

### 開発者向けチェックリスト

1. 新エディタモード追加時は **`suspend()` を必ず実装**（タイマー・listen・Rust セッションの漏れ防止）
2. ウィンドウ close で **`deactivate()` だけ** にしない — suspend 専用パスを使う
3. トレイから開くファイルは **`editorModeForPath`** でモード合わせ
4. バックグラウンド処理は suspend 後に **協調的キャンセル**（CSV の `SESSION_EPOCH` パターン）
5. トレイメニュー項目変更時は **`build_tray_menu` / `refresh_tray_menu`** を同期
6. 作業フォルダの **ディスク永続化**（`workspace.json`）と **メモリ上の `workspaceRoot`** を混同しない
7. 新しい **フロント window API**（`hide`, `setTitle` 等）追加時は [`capabilities/default.json`](../src-tauri/capabilities/default.json) に allow があるか確認

### スコープ外（再提案しない）

| 案 | 理由 |
| -- | ---- |
| 閉じるたび WebView 破棄・再生成 | 再表示レイテンシ・実装コスト |
| `LSUIElement`（Dock 非表示） | ユーザーが Dock からも起動できる現状を維持 |
| トレイに CSV 索引進捗常時表示 | 待機中 CPU/描画コスト。ウィンドウ表示中の UI で足りる |
| バナー通知（Notification Center） | Status Item とは別 API。本機能の範囲外 |

## 表示設定（フォント・サイズ）

### なぜ Web だけではフォントを列挙できないか

ブラウザ / WebView は **OS にインストールされたフォント名の一覧を標準 API では返しません**。  
そのため初期実装では「よくあるフォント名」のプリセットだけを用意していました。

### 現在の方式

| 候補                                              | 採用                     | 理由                                                                                    |
| ------------------------------------------------- | ------------------------ | --------------------------------------------------------------------------------------- |
| 固定プリセットのみ                                | フォールバックとして残す | Tauri 外の `vite` 単体開発時用                                                          |
| **Rust `fontdb` でシステムフォント列挙**          | **採用**                 | 実際に入っているフォントだけを設定に出す。設定モーダルを開いたとき（＋起動後 BG）に取得 |
| フォント変更のたび Web Font 読み込み              | 却下                     | I/O とメモリ。採用しない                                                                |
| **CSS 変数で `font-family` / `font-size` を切替** | **採用**                 | Markdown の再パース不要。ランタイムコストは変更時の 1 回レイアウト程度                  |
| 設定 UI をツールバー下に常時表示                  | 却下                     | 画面を圧迫                                                                              |
| **設定はモーダル**                                | **採用**                 | 必要なときだけ                                                                          |
| **エディタ常時ライト（オプション）**              | **採用**（C3）           | `:root.editor-light` でダーク OS でも編集面をライト固定。既定 OFF                       |
| **作業フォルダのファイル監視（オプション）**      | **採用**（B1）           | `notify` + 300ms デバウンス。設定で ON、**既定 OFF**                                    |

編集用リストは等幅フォントを中心に、プレビューは非等幅を中心に UI で分けています（`src/settings.ts`）。表示設定は `localStorage` の `karasu-display-settings` に保存。

## シンタックスハイライト

詳細は [`syntax-highlighting.md`](syntax-highlighting.md) に記載。

## Tauri 設定を極限まで削らない理由

`core:default` や `dialog:default` のまま、Cargo の `tauri` feature も広めに残しています。

| 検討                              | 結論                                                                  |
| --------------------------------- | --------------------------------------------------------------------- |
| capabilities を最小権限だけに削る | 可能だが **体感性能はほぼ変わらない**（主にバイナリと ACL）           |
| WebView をやめる                  | プレビュー HTML のため現実的でない                                    |
| **現状維持**                      | **開発コスト対効果**が小さい。ボトルネックは WebView とプレビュー DOM |

将来、配布サイズや攻撃面を詰めるフェーズで見直す余地はあります。

### バックログの位置づけ

| 観点   | 説明                                                                                                  |
| ------ | ----------------------------------------------------------------------------------------------------- |
| 目的   | 多数ファイルの切り替え・日常操作の改善を、**軽量方針を壊さない範囲**で進める                          |
| 非目的 | Obsidian / VS Code の全機能再現、ノート DB、Git UI、WYSIWYG                                           |
| 評価軸 | [設計で最優先していること](#設計で最優先していること) の順（CPU → メモリ → GPU → I/O → ビルドサイズ） |

### 現状のボトルネック（比較の基準）

拡張案を評価するときは、次を「すでにかかっているコスト」として意識する。

| 要因                      | 待機中             | スパイクが出る操作                 | 備考                             |
| ------------------------- | ------------------ | ---------------------------------- | -------------------------------- |
| **WKWebView**             | メモリ・GPU の大半 | プレビュー HTML の描画・スクロール | 回避不可（プレビュー方針の前提） |
| **`marked` + sugar-high** | 影響なし           | プレビュー切替時（キャッシュあり） | 編集中はゼロ                     |
| **Mermaid（遅延 import）** | 影響なし           | 図あり MD の初回プレビューでチャンク取得 | 編集中はゼロ              |
| **`fontdb` 列挙**         | 影響なし           | 設定モーダル初回・起動後 BG 1 回   | 常駐しない                       |
| **サイドバー遅延ツリー**  | 影響なし           | フォルダ展開・↻ 再読み込み         | 展開分だけ DOM                   |
| **明示保存・`read_file`** | 影響なし           | ファイルを開く・保存したとき       | 既存コア                         |

バックログの多くは **この上に乗る追加分** であり、◎〜○なら「採用しても芯の軽さは保ちやすい」という意味になる。

### 運用ルール（バックログ更新時）

1. 実装をマージしたら、該当 ID の行に **ステータス `実装済み`** と PR / コミットを追記する。
2. 却下したら **E 節** に移し、却下日と理由を 1 行足す。
3. 新案を足すときは **ID を採番**（F1, F2…）し、最低限「詳細・影響・難度・工数」の列を埋める。
4. 性能に関わる変更は、可能なら **S/M/L 規模** で手動確認結果を 1 行メモする。
